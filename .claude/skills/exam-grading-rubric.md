# Exam Grading Rubric

iOS 면접 주관식 답안을 4개 기준 × 0/1점으로 점검해 **0~5점 총점**과 **개선 코멘트**를 즉시 산출한다. `/admin/grading` 화면의 자기 점검 4문항을 그대로 코드화한 것이다.

## 4 기준 (각 1점, 총 4점)

| # | 기준 | 통과 판정 |
|---|---|---|
| C1 | **결론 우선** | 답변 첫 1~2문장이 *목적/결론*을 명시했는가. "이건 X를 해결하려는 Y다" 형태로 시작했는지 |
| C2 | **메커니즘** | 내부 동작/비용/시점 중 *최소 한 가지*를 한 단계 이상 설명했는가. "왜 그렇게 되는가"가 있는지 |
| C3 | **구체 예시** | 코드 한 토막 / 실무 상황 / 구체 시나리오 중 하나가 있는가. *추상 어휘만 반복*하지 않았는지 |
| C4 | **트레이드오프·함정** | 한계, 대안, 흔한 실수, *언제 쓰면 안 되는가*가 언급됐는지 |

C1~C4 합계 + **명료성 보너스 1점 = 0~5점 최종**.

명료성 보너스 기준:
- 답변이 짧지만 군더더기 없이 핵심에 닿았다 (+1)
- 답변이 길지만 흐름이 또렷하고 면접 현장에서 1분 안에 말할 수 있다 (+1)
- 그 외 → 보너스 없음

## 채점 출력 포맷

채점 요청을 받으면 반드시 아래 형식 그대로 응답한다 (군더더기 금지):

```
## 채점: X/5

| 기준 | 통과 | 근거 |
|---|---|---|
| C1 결론 우선     | ✅/❌ | (답변 어느 부분이 어떤 이유로 통과/탈락) |
| C2 메커니즘       | ✅/❌ | ... |
| C3 구체 예시      | ✅/❌ | ... |
| C4 트레이드오프   | ✅/❌ | ... |
| 명료성 보너스     | +0/+1 | ... |

### 한 줄 평
(이 답변의 본질적 약점 또는 강점 한 줄)

### 다음 시도에 고칠 한 가지
(가장 ROI 높은 1개만. 두 개 이상 쓰지 말 것)

### 모범 답안과의 결정적 차이 (있을 때만)
- ...
```

## 채점 원칙

- **인플레이션 금지**: C2~C4 미달이면 **2점 이상 주지 않는다**. "사실관계는 맞으니 3점" 같은 후한 채점이 가장 흔한 오류
- **암기형 vs 이해형**: Apple Docs 어휘를 늘어놓는데 *왜*가 빠지면 C2 탈락
- **분량으로 점수 안 줌**: 길이가 길어도 C1/C3가 없으면 그대로 탈락
- **모범 답안과 비교는 마지막에**: 우선 답변 자체로만 채점한 뒤, 결정적 차이가 있을 때만 마지막 섹션에 한두 줄 추가
- **5/5는 아끼기**: 면접 자리에서 그대로 말해도 통과할 수준일 때만. 학습 단계에서 4/5가 일반적

## 입력 처리

다음 중 어떤 형태로 들어와도 동일 포맷으로 응답한다:

1. **"이 답안 채점해줘"** + [질문]/[답변] 텍스트
2. **`/admin/grading` 페이지의 카드 한 장을 통째로 붙여넣음**
3. **답변만** 줬을 때 — 질문을 추정해 적고, *추정한 질문이 맞는지* 한 줄로 먼저 확인하고 채점
4. 여러 문항을 한꺼번에 → 각 문항별로 위 포맷을 반복

## 모범 답안 참조

해당 토픽 문서가 `docs/`에 있으면 *읽고 비교*한다:

- `docs/01-swift-language/` ~ `docs/14-network/` 의 답변 문서가 1차 비교 기준
- 문서의 "한 줄 요약 / 비교표 / 흔한 함정" 섹션이 C1/C4 판정의 직접 근거
- 문서가 없는 토픽이면 "모범 답안과의 결정적 차이" 섹션을 비우고, 한 줄 평에 *문서 누락*을 명시

## DB 반영 (`/admin/grading` 사용 시)

수동 채점 화면(`/admin/grading`)에 입력할 값:

- **grade 라디오**: 위에서 산출한 0~5점
- **note**: `### 한 줄 평`과 `### 다음 시도에 고칠 한 가지` 두 줄만 압축해서 붙여넣기 (전체 표는 메모로는 과함)

## 예시

**입력:**

> 질문: weak와 unowned의 차이는?
> 답변: weak는 옵셔널이고 unowned는 비옵셔널이에요. weak가 더 안전합니다.

**출력:**

## 채점: 1/5

| 기준 | 통과 | 근거 |
|---|---|---|
| C1 결론 우선     | ❌ | "X를 해결하기 위한 Y" 구조 없이 표면 차이만 나열 |
| C2 메커니즘       | ❌ | side table, retain count 미증가, 해제 후 동작 등 *왜* 그렇게 되는지 부재 |
| C3 구체 예시      | ❌ | 코드도 시나리오도 없음 |
| C4 트레이드오프   | ❌ | unowned가 왜 위험한지, 언제 unowned가 정당한지 미언급 |
| 명료성 보너스     | +1 | 짧고 또렷함 |

### 한 줄 평
사실관계는 맞지만 *왜 두 종류가 따로 존재하는가*에 대한 설명이 0이라 면접에선 즉시 추가 질문이 들어옴.

### 다음 시도에 고칠 한 가지
"상대가 self보다 먼저 사라질 가능성이 있느냐"는 *선택 기준 한 문장*을 결론으로 먼저 말하기.

### 모범 답안과의 결정적 차이
- side table을 통한 자동 nil 처리 vs 해제 후 접근 시 크래시라는 동작 메커니즘 누락 ([docs/02-memory-management/weak-vs-unowned.md](../../docs/02-memory-management/weak-vs-unowned.md))
